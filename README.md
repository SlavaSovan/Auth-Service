# Управление доступом в Auth service
## Общая концепция
Система управления доступом построена на основе ролевой модели (RBAC - Role-Based Access Control) с гранулярными разрешениями. Это обеспечивает гибкое и безопасное разграничение прав доступа пользователей к различным ресурсам и операциям в системе.

## Пользователи
Каждый пользователь обладает уникальным идентификатором, электронной почтой для аутентификации, персональными данными (ФИО) и привязан к определенной роли, которая определяет набор его прав. Пользователи могут быть активными или неактивными — последние не могут проходить аутентификацию и взаимодействовать с системой.

## Роли
Роли представляют собой именованные наборы прав, объединенные по функциональному признаку. Каждая роль имеет уникальное название, описательное назначение и флаг, указывающий, является ли она ролью по умолчанию для новых пользователей. Система поддерживает предопределенные роли: администратор с полным доступом ко всем ресурсам и стандартный пользователь с базовыми правами для работы с собственными данными.

## Разрешения
Разрешения являются атомарными единицами прав доступа. Каждое разрешение имеет уникальное имя, описание и категорию для логической группировки. Разрешения независимы от ролей и могут быть скомбинированы в любом наборе при создании или настройке роли.


# Процесс проверки прав доступа
## 1. Аутентификация пользователя
Первый этап — проверка подлинности пользователя. При обращении к защищенным ресурсам система извлекает JWT-токен из запроса, проверяет его валидность, срок действия и отсутствие в черном списке отозванных токенов. После успешной валидации из токена извлекается идентификатор пользователя, по которому загружается полная информация о пользователе.

## 2. Загрузка прав доступа
После аутентификации система выполняет "жадную" загрузку всех связанных с пользователем данных: его роль и все разрешения, привязанные к этой роли. Это позволяет выполнять проверку прав без дополнительных запросов к базе данных в процессе обработки запроса.

## 3. Проверка разрешений
Непосредственная проверка доступа может выполняться несколькими способами. В рамках сервиса используется проверяющий класс PermissionChecker, который инкапсулирует логику проверки конкретного разрешения и может быть легко подключен к эндпоинту через зависимости в FastAPI. Для работы с вымышленными объектами бизнес-приложения логика проверки прав доступа выполнялась непосредственно в коде эндпоинта.

# Управление ролями и разрешениями
## Создание инастройка ролей
Администраторы могут создавать новые роли, задавать им название и описание, а также определять набор разрешений. При создании роли можно указать, должна ли она стать ролью по умолчанию для новых пользователей. Система автоматически следит, чтобы только одна роль могла быть ролью по умолчанию.

## Назначение разрешений ролям
Управление разрешениями ролей реализовано через специализированные методы, позволяющие добавлять и удалять отдельные разрешения, а также полностью заменять набор разрешений роли. Это дает возможность точно настроить права для каждой роли.

## Проверка зависимостей
Перед удалением разрешения система проверяет, не используется ли оно в какой-либо роли. Если разрешение привязано к ролям, его удаление блокируется, предотвращая потерю целостности прав доступа. Аналогично, перед удалением роли проверяется, не привязаны ли к ней пользователи — роль с пользователями не может быть удалена.


# Инструкция по установке и настройке Auth Service
## Системные требования
* **Docker:** версия 20.10 или выше
* **Docker compose**: версия 2.0 или выше
* **Git:** для клонирования репозитория
* **Операционная система:** Linux, macOS или Windows (с WSL2)
* **Свободное место:** не менее 1 ГБ
* **Порты:** 8000, 5050 должны быть свободны

## Установка
### Шаг 1: Клонирование репозитория
```
git clone https://github.com/SlavaSovan/Auth-Service.git
```
### Шаг 2: Переход в директорию проекта
```
cd auth-service
```
### Шаг 3: Настройка переменных окружения
Необходимо создать файл `.env` в корневой директории проекта и задать необходимые параметры:
```
DB_NAME=auth
DB_USER=admin
DB_PASSWORD=admin

PGADMIN_EMAIL=admin@admin.com
PGADMIN_PASSWORD=admin
```

### Шаг 4: Запуск контейнеров
Если установлена утилита make:
```
make up
```
Если утилита make не установлена:
```
docker-compose up -d
```
Также можно запустить систему в режиме разработки.
Если установлена утилита make:
```
make up-dev
```
Если утилита make не установлена:
```
docker-compose up --watch
```
### Шаг 5: Проверка статусов контейнеров
Если установлена утилита make:
```
make ps
```
Если утилита make не установлена:
```
docker-compose ps
```

## Инициализация системы
### Шаг 1: Инициализация ролей и разрешений
Если установлена утилита make:
```
make init
```
Если утилита make не установлена:
```
docker exec -it auth_service python -m app.scripts.init_roles
```
Система автоматически создаст базовые разрешения для работы как с самим сервисом, так и с вымышленными объектами бизнес-приложения. Также будут созданы роли user и admin
### Шаг 2: Создание администратора
Если установлена утилита make:
```
make create-admin
```
Если утилита make не установлена:
```
docker exec -it auth_service python -m app.scripts.create_admin
```
## Документация API
Интерактивная документация доступна по адресу: http://localhost:8000/api/v1

## Доступ к PgAdmin
Управления базой данных через веб-интерфейс доступно по адресу: http://localhost:5050. Для входа использовать данные, указанные в `.env`. При добавлении сервера указывать хост `pg`.
